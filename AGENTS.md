# AGENTS.md

## Apple / Xcode Project Workflow

- Use `xcode-build-run-workflow` for normal Xcode build, run, diagnostics, preview, file-membership, and guarded mutation work inside this existing project.
- Use `xcode-testing-workflow` when the task is primarily about Swift Testing, XCTest, XCUITest, `.xctestplan`, flaky tests, retries, or test diagnosis.
- Use `apple-ui-accessibility-workflow` when the task is primarily about SwiftUI accessibility semantics, Apple UI accessibility review, accessibility tree shaping, or UIKit/AppKit accessibility bridge behavior.
- Use `sync-xcode-project-guidance` when the repo guidance for this project drifts and needs to be refreshed or merged forward.
- Use `scripts/repo-maintenance/validate-all.sh` for local maintainer validation, `scripts/repo-maintenance/sync-shared.sh` for repo-local sync steps, and `scripts/repo-maintenance/release.sh` for releases.
- Read relevant Apple documentation before proposing or making Xcode, SwiftUI, lifecycle, architecture, or build-configuration changes.
- Before doing any SwiftUI, scene, command, focus, or close-behavior work in this repository, read `docs/maintainers/workspace-focus-guide.md` first. Treat that maintainer doc as the repo's current default guidance and implementation map, then re-check the relevant Apple documentation when a design decision or tradeoff is unclear.
- For SwiftUI command, focus, menu, window, and scene research in this repo, prefer the Xcode MCP `DocumentationSearch` tool first. Use Dash as a fallback or cross-check, and on this machine expect the most relevant installed Apple docsets to be the generic Swift API reference (`ntiaiyxj-swift`), Objective-C API reference (`ntiaiyxj-objc`), and offline macOS set (`jtswqsfb`) rather than a dedicated SwiftUI docset.
- Prefer Dash or local Apple docs first, then official Apple docs when local docs are insufficient.
- Prefer the simplest correct Swift that is easiest to read and reason about.
- Prefer synthesized and framework-provided behavior over extra wrappers and boilerplate.
- When Gale explicitly asks for a simplification pass, treat aggressive code reduction as the primary directive: remove redundant layers, helper functions, wrappers, indirection, duplicate symbols, convenience abstractions, and boilerplate until each remaining symbol has a concrete job that cannot be expressed more directly with existing language or framework features. Do that work in place without hiding code by moving it into new files or new wrapper types.
- Default to SwiftUI and AppKit built-ins whenever they already model the behavior we need. Do not replace framework-provided scene routing, focus handling, menu behavior, window behavior, responder-chain behavior, sheet dismissal, toolbar behavior, or event handling with custom in-house infrastructure unless there is a concrete product requirement the frameworks cannot satisfy cleanly.
- Prefer standard SwiftUI command groups and documented `CommandGroup` insertion or replacement over hand-built copies of framework command behavior. When `SidebarCommands`, `InspectorCommands`, `ToolbarCommands`, or another built-in command group matches the product surface, use that built-in group first and customize it through SwiftUI's documented command APIs rather than reimplementing the command set from scratch.
- For command architecture specifically, use `docs/maintainers/workspace-focus-guide.md` as the repo's strong default rather than as infallible law. Prefer scene commands on scenes, focused values for command context, environment for downward configuration, and preferences for upward child-to-container signals unless current Apple documentation or a concrete product need points to a better fit.
- Do not invent custom routing systems, pseudo-responder layers, shadow selection models, custom close-command pipelines, or private command buses to work around SwiftUI or AppKit unless you can point to the relevant Apple API behavior, explain why it is insufficient here, and get Gale's approval for the tradeoff.
- Do not manually reroute close commands through app-defined command infrastructure when the standard SwiftUI or AppKit dismissal path already applies. Presented views should normally dismiss through the presenting context that owns them, and ordinary windows should normally close through the framework's standard close flow.
- Treat custom close-command handling as an exception. If a custom close path is still required, keep it narrow, local to the unsupported presentation or window type, and isolated from unrelated scene, model, or command-selection logic.
- Before introducing custom command routing, manual event handling, bespoke focus systems, custom window or sheet management, or other framework-overriding infrastructure, explicitly document the built-in SwiftUI or AppKit behavior being declined, the concrete gap it fails to cover, the near-term use case unlocked by the custom path, and the maintenance cost being accepted. Gale must approve that tradeoff before implementation.
- If AppKit already provides the correct primitive through views, view controllers, window delegates, the responder chain, menu validation, or application lifecycle hooks, prefer adopting that AppKit surface over inventing parallel lightweight abstractions in SwiftUI-only code.
- Keep data flow straight and dependency direction unidirectional.
- Treat the `.xcworkspace` or `.xcodeproj` as the source of truth for app integration, schemes, and build settings.
- Prefer Xcode-aware tooling or `xcodebuild` over ad hoc filesystem assumptions when project structure or target membership is involved.
- Never edit `.pbxproj` files directly. If a project-file change is needed and no safe project-aware tool is available, stop and make that change through Xcode instead.
- Validate Xcode-project changes with explicit `xcodebuild` commands when build or test integrity matters.
