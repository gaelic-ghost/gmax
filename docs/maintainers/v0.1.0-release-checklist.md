# v0.1.0 Release Checklist

This checklist is for the first usable internal `gmax` build. It is intentionally product-oriented: the goal is to verify that the current shell feels coherent, predictable, and debuggable before cutting `v0.1.0`.

## Versioning And Release Surface

- [ ] Set the app target `MARKETING_VERSION` to `0.1.0` everywhere the main app target defines it.
- [ ] Confirm the release tag will be `v0.1.0` so the repo-maintenance release flow and the app version agree.
- [ ] Review `README.md`, `ROADMAP.md`, and `TODO.md` together so the docs describe the same current product state.

## Build And Automated Verification

- [ ] Run `xcodebuild -project gmax.xcodeproj -scheme gmax -destination 'platform=macOS' build`.
- [ ] Run `xcodebuild -project gmax.xcodeproj -scheme gmax -destination 'platform=macOS' test`.
- [ ] Record the current expected skipped tests, if any, so template scaffolding does not get mistaken for release breakage.

## Manual Product Verification

- [ ] Launch the app from Xcode and confirm the default window opens with sidebar, content, and inspector visible.
- [ ] Create, rename, duplicate, close, reopen, and delete workspaces without losing selection or leaving the UI in an ambiguous state.
- [ ] Create panes, split right and down, move pane focus in every supported direction, and close panes until the last-pane confirmation path appears.
- [ ] Save a workspace to the library, reopen it, and confirm pane layout plus transcript-backed shell history are restored.
- [ ] Verify the saved-workspace library supports keyboard selection, default-action open, and clear empty-state messaging.
- [ ] Change terminal appearance settings and confirm the live terminal surfaces update accordingly.

## Accessibility And Keyboard Workflow

- [ ] Review [accessibility-and-keyboard-plan.md](accessibility-and-keyboard-plan.md) before the manual accessibility pass so known risks and acceptable `v0.1.0` limitations are explicit.
- [ ] Verify every primary action in the sidebar, toolbar, pane tree, inspector, and saved-workspace sheet is reachable without a mouse.
- [ ] Confirm the currently focused pane is visually obvious and that focus changes are predictable after splits, closes, reopen, and relaunch.
- [ ] Run a VoiceOver pass across the main shell and note where pane metadata, terminal state, or custom AppKit-hosted content are missing or unclear.
- [ ] Run a Full Keyboard Access pass and note any controls that cannot be reached or activated reliably.

## Operator-Facing Quality

- [ ] Exercise workspace restore, shell relaunch, and saved-workspace reopen flows while watching logs for vague or context-free messages.
- [ ] Confirm that user-visible alerts and confirmations describe what happened, what will close, and what state will remain.
- [ ] Review persistence and restore failures for actionable wording before calling the build release-ready.

## Release Execution

- [ ] Run `scripts/repo-maintenance/validate-all.sh`.
- [ ] Run `scripts/repo-maintenance/release.sh --version v0.1.0` once the version, docs, and verification surface are ready.
- [ ] Capture release notes that summarize shell capabilities, known accessibility gaps, and any intentionally deferred product work.
