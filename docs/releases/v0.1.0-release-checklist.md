# v0.1.0 Release Checklist

This checklist is for the first usable internal `gmax` build. It is intentionally product-oriented: the goal is to verify that the current shell feels coherent, predictable, and debuggable before cutting `v0.1.0`.

## Versioning And Release Surface

- [ ] Set the app target `MARKETING_VERSION` to `0.1.0` everywhere the main app target defines it.
- [ ] Confirm the release tag will be `v0.1.0` so the repo-maintenance release flow and the app version agree.
- [ ] Review `README.md`, `ROADMAP.md`, and `TODO.md` together so the docs describe the same current product state.
- [ ] Confirm the maintainer notes under `docs/maintainers/` still match the current source and test organization before cutting the release.

## Build And Automated Verification

- [ ] Run `xcodebuild -project gmax.xcodeproj -scheme gmax -destination 'platform=macOS' build`.
- [ ] Run `xcodebuild -project gmax.xcodeproj -scheme gmax -destination 'platform=macOS' test`.
- [ ] Record the current expected skipped tests, including the intentionally disabled launch placeholder, so template scaffolding does not get mistaken for release breakage.
- [ ] Confirm the domain-grouped unit-test files still map cleanly to current source ownership so future release fixes do not collapse back into a grab-bag test file.
- [ ] Confirm the grouped UI-test files still map cleanly to current sidebar, library, and launch ownership instead of drifting back into one generic UI suite.
- [ ] Confirm the persistence suite covers both successful restore flows and failure modes such as corrupted pane trees, missing pane session snapshots, and other partial restore inputs.

## Manual Product Verification

- [ ] Launch the app from Xcode and confirm the default window opens with sidebar, content, and inspector visible.
- [ ] Open a second shell window and confirm workspace, pane, save, and delete commands follow the frontmost window's scene-local selection instead of mutating the background window.
- [ ] Create, rename, duplicate, close, reopen, and delete workspaces without losing selection or leaving the UI in an ambiguous state.
- [ ] Create panes, split right and down, move pane focus in every supported direction, and close panes until the last-pane confirmation path appears.
- [ ] Save a workspace to the library, reopen it, and confirm pane layout plus restored terminal history are restored.
- [ ] During that reopen check, confirm ordinary shell panes reopen near the prior scroll position when saved history included one.
- [ ] Verify the saved-workspace library supports keyboard selection, default-action open, and clear empty-state messaging.
- [ ] Change terminal appearance settings and confirm the live terminal surfaces update accordingly.

## Accessibility And Keyboard Workflow

- [ ] Review [accessibility-and-keyboard-plan.md](accessibility-and-keyboard-plan.md) before the manual accessibility pass so known risks and acceptable `v0.1.0` limitations are explicit.
- [ ] Verify every primary action in the sidebar, toolbar, pane tree, inspector, and saved-workspace sheet is reachable without a mouse.
- [ ] Confirm the currently focused pane is visually obvious and that focus changes are predictable after splits, closes, reopen, and relaunch.
- [ ] Run a VoiceOver pass across the main shell and note where pane metadata, terminal state, or custom AppKit-hosted content are missing or unclear.
- [ ] Run a Full Keyboard Access pass and note any controls that cannot be reached or activated reliably.
- [ ] Record the exact macOS version, VoiceOver state, and Full Keyboard Access state used for the manual pass so later regressions are comparable.
- [ ] Sidebar pass:
  - [ ] Move through the workspace list with the keyboard only.
  - [ ] Confirm the selected workspace is obvious visually and audibly.
  - [ ] Open rename and delete flows without a mouse and verify the target workspace name is spoken clearly in the sheet or alert copy.
- [ ] Toolbar and command pass:
  - [ ] Reach toolbar actions without a mouse.
  - [ ] Confirm menu titles, toolbar labels, and spoken action names use the same vocabulary for workspace, pane, and inspector actions.
  - [ ] Confirm the pane-creation vocabulary stays split-based in both toolbar buttons and the `New Pane` menu section instead of reviving a generic `New Pane` action.
  - [ ] Verify the saved-workspace library, settings window, and close flows remain reachable even when pane focus has recently moved through the terminal host.
- [ ] Pane-tree pass:
  - [ ] Move focus between panes with the supported keyboard commands and confirm the visible focus treatment follows the actual focused pane.
  - [ ] With VoiceOver enabled, confirm each pane reads as a stable control with a useful label, value, and available actions.
  - [ ] Invoke pane-local actions from the accessibility surface and verify they perform the same model operation as the pointer path.
  - [ ] Confirm split dividers expose adjustable behavior or, if that remains insufficient, note the exact limitation instead of leaving it implicit.
- [ ] Inspector pass:
  - [ ] Confirm the inspector remains a metadata-only surface and does not duplicate primary pane actions that already live in the toolbar or menus.
  - [ ] Confirm metadata reads as grouped information rather than an unordered wall of text.
  - [ ] Check that debugging identifiers do not dominate the spoken reading order ahead of pane state and actions.
- [ ] Saved-workspace library pass:
  - [ ] Navigate the library list and search field using only the keyboard.
  - [ ] Open a saved workspace with the default keyboard action.
  - [ ] Verify empty-state and error-state messaging describe what the user can do next.
- [ ] Embedded terminal host pass:
  - [ ] Enter the terminal host from the pane tree and return to adjacent shell chrome without getting trapped.
  - [ ] Note whether VoiceOver can identify the host container as the active pane even when live terminal content is limited.
  - [ ] Note whether Full Keyboard Access can continue past the host view to toolbar, sidebar, inspector, and sheet controls.
  - [ ] Capture any live-terminal gaps separately from shell-level pane metadata gaps so `gmax` issues and SwiftTerm issues do not get conflated.
- [ ] End the pass with a short findings note that separates:
  - [ ] fixed-enough-for-`v0.1.0` limitations
  - [ ] `gmax` regressions that must be repaired before release
  - [ ] likely SwiftTerm upstream issues tracked by `Milestone 3A`

## Operator-Facing Quality

- [ ] Exercise workspace restore, shell relaunch, and saved-workspace reopen flows while watching logs for vague or context-free messages.
- [ ] Use [logging-validation-guide.md](logging-validation-guide.md) during manual validation so Console filters, `/usr/bin/log` commands, and expected test noise are consistent across maintainers.
- [ ] Confirm that user-visible alerts and confirmations describe what happened, what will close, and what state will remain.
- [ ] Review persistence and restore failures for actionable wording before calling the build release-ready.
- [ ] Verify persistence and restore failures degrade safely: the app should keep the live session usable, skip only the damaged data it cannot recover, and avoid ambiguous half-restored shell state.
- [ ] Confirm logs and operator wording stay honest about the current terminal restore scope: restored history and ordinary scroll position, not full alternate-buffer or TUI replay.
- [ ] Review [logging-and-telemetry-options.md](logging-and-telemetry-options.md) before changing the logging surface so `v0.1.0` keeps one clear baseline for operator logs, diagnostics capture, and future support bundles.

## Release Execution

- [ ] Run `scripts/repo-maintenance/validate-all.sh`.
- [ ] Run `scripts/repo-maintenance/release.sh --version v0.1.0` once the version, docs, and verification surface are ready.
- [ ] Capture release notes that summarize shell capabilities, known accessibility gaps, and any intentionally deferred product work.
